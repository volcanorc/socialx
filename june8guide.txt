Original problem
The user reported that records created while logged in with the same Google account were not appearing across devices, even though local and production were both connected to the same Neon backend. The concern was that the app might be:
- using different backend URLs in different environments,
- assigning different auth user IDs for the same Google account,
- saving data only to localStorage instead of Neon,
- filtering reads by the wrong owner key,
- or creating duplicate owner/user rows during auth bootstrap.

The user explicitly asked for proof first, then a production-grade fix that guarantees one Google account maps to one shared user identity and shared data across devices.

High-level conclusion
The codebase confirmed a single Neon Auth URL and a single Neon Data API URL. The primary cause was not multiple database URLs. The main issue was that the app was treating the raw Neon auth user ID as the owner identity everywhere, while localStorage remained a usable fallback snapshot. There was no canonical identity resolver in the app or schema before this work.

What was already true before this investigation
1. Single backend wiring
- The app used a single hardcoded Neon Auth URL and Neon Data API URL in:
  - index.html
  - src/config.js
  - src/neon.js
- server.mjs was only a static server and did not override backend URLs.

2. Owner-keyed data model existed
- Data tables were already keyed by owner_auth_user_id in src/store.js and sql/schema.sql.
- The visible data model was owner-based, but the owner key was still just the auth user ID.

3. localStorage fallback existed
- src/store.js used localStorage key socialx:owners:v2 as a snapshot cache.
- sessionStorage was only used for the vault passphrase.
- The app could therefore appear to have complete data on one device even if Neon sync did not actually succeed.

4. No provider-subject canonical identity code existed
- Searches of src/main.js, src/store.js, src/auth.js, and sql/schema.sql found no code that extracted or persisted a Google provider subject / sub claim.
- The app had no notion of a canonical Google identity separate from the Neon auth user ID.

Evidence gathered
Code-path evidence:
- src/main.js set state.ownerId directly from result.user?.id / result.session?.userId / result.session?.user?.id.
- src/main.js passed state.ownerId into store.initialize(), store.listAccounts(), store.createAccount(), store.updateAccount(), store.archiveAccount(), store.deleteAccount(), store.importOwner(), and store.syncOwner().
- src/store.js loaded remote data by filtering Neon tables on auth_user_id / owner_auth_user_id equal to the raw owner ID.
- src/store.js flushed remote data by deleting and reinserting rows for that raw owner ID.
- sql/schema.sql enforced RLS and ownership based on auth.user_id() = auth_user_id or owner_auth_user_id before the canonical identity change.

Runtime behavior evidence:
- renderSyncStatus(owner) already surfaced whether Neon sync had errors.
- The UI had no identity debug surface, so the exact auth/session/user-claim state was not visible before this investigation.

Proved hypotheses
1. Disproven: “There are multiple backend URLs in the source tree.”
- The code consistently used the same Neon Auth and Data API URLs in the app bundle and runtime config.

2. Proven: “The app was using the raw auth user ID as the owner identity.”
- state.ownerId came directly from the Neon auth session user ID fields.
- The store and all reads/writes were keyed to that value.

3. Proven: “localStorage could make one device look complete even when Neon sync failed.”
- The store had a local snapshot cache and would use it when remote loading was unavailable.
- Sync failures were tracked, but the UI did not show enough identity detail to disambiguate a local-only snapshot from a remote one.

4. Partially proven: “Same Google email may not equal same Neon auth user ID across devices.”
- The repo could not prove live session payloads across devices without runtime inspection, but the architecture strongly indicated that data sharing depended on the raw Neon auth ID.
- Because no canonical provider subject mapping existed, any auth-user-ID drift would split data across devices.

What changed during the investigation

1. Added canonical identity resolution plumbing
Files:
- src/store.js
- sql/schema.sql
- src/main.js

Reason:
- To stop treating the raw auth user ID as the durable owner identity.
- To create a stable owner mapping path that can survive across devices and across auth-session differences.

What was added:
- A new database function in sql/schema.sql:
  - resolve_owner_identity(...)
- A new mapping table in sql/schema.sql:
  - owner_auth_links
- New canonical columns on users:
  - canonical_key
  - google_subject
  - google_email

Why:
- The app needs a stable owner row that can be resolved from the same Google identity across devices.
- The app needed a bootstrap-safe place to map auth user ID -> canonical owner ID.

2. Updated Neon RLS and grants to use owner links
File:
- sql/schema.sql

Reason:
- To let authenticated users access the canonical owner rows linked to their current auth identity.

What changed:
- Policies for users, accounts, account_relationships, custom_fields, custom_field_values, and activity_log were changed to check owner_auth_links instead of direct auth.user_id() equality.
- owner_auth_links got explicit grants.
- The resolver function got an execute grant for authenticated users only.

Why:
- The old RLS depended directly on auth.user_id() and could not represent a canonical shared owner across devices.
- The new design requires the auth user to be linked to the canonical owner row.

Known limitation:
- The schema now relies on a bootstrap-safe resolver function to create or link the owner mapping before the app reads/writes data.
- This design was intentional, but it is the main place the next AI should continue validating for edge cases.

3. Added store-side canonical identity resolution
File:
- src/store.js

Functions added:
- normalizeIdentityKey(value)
- mergeCustomPlatforms(base, incoming)
- mergeOwnerSnapshots(baseOwner, incomingSnapshot)
- resolveOwnerIdentity(identity)
- resetMemory()

Functions updated:
- createEmptyOwnerState(ownerId)
- rowToUser(row)
- userToRow(ownerId, snapshot)
- normalizeOwnerSnapshot(ownerId, snapshot)
- loadRemoteOwner(ownerId)
- flushOwner(ownerId)
- initialize(ownerId, profile, options)

Reason:
- The app needed to resolve a canonical owner before loading/saving data.
- The app needed to merge old local snapshots into the canonical owner rather than leaving duplicate owner islands.
- The app needed to preserve local cache as fallback only, not as the authoritative shared dataset.

What changed:
- createEmptyOwnerState now stores canonical profile metadata and sync source.
- rowToUser/userToRow now round-trip canonical_key, google_subject, and google_email.
- normalizeOwnerSnapshot now preserves canonical metadata and normalizes owner-owned child records.
- mergeOwnerSnapshots merges local cache snapshots into a single owner state.
- resolveOwnerIdentity calls the Neon RPC when available and returns a canonical owner mapping.
- initialize now accepts aliases and merges any matching local snapshots into the canonical owner.
- flushOwner now updates the canonical owner and stops deleting the owner row on every sync.
- resetMemory clears in-memory owner caches on sign-out.

Why:
- Without this, the same Google account could produce separate device-specific owner islands.
- The app had to distinguish between raw auth ID and canonical owner ID.
- The local cache had to remain a fallback, but not the source of truth for cross-device identity.

Important implementation note:
- The store now treats the auth user ID as the linked auth identity and the canonical key as the durable owner identity.
- Remote loading/saving still uses Neon Data API, but the owner key is no longer assumed to be a direct auth-user-ID-only singleton.
